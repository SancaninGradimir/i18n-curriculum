# --description--

Jesi li se ikada pitali kako da vašu Express aplikaciju održite stabilnom i spremnom za produkciju, posebno kada je pod velikim opterećenjem ili se ga isključuje?

Uloga ovde imaju provere zdravlja i graciozni gašenja. Ova dva obrasca pomažu vam da održite dostupnost vaše aplikacije i osiguraju da napušta čistim putem bez gubljenja zahteva ili korumpiranja podataka.

Razlojićemo to.

Prvo, šta je uopšte „provera zdravlja“ u Expressu?

Provera zdravlja je jednostavan rute koji govori da li vaša aplikacija radi i funkcioniše.

Na primer:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Alati kao što su balanseri opterećenja ili orkestratori kontejnera – razmislite o Kubernetesu ili AWS Elastic Beanstalk-u – mogu da pinge ovu rutu kako bi odlučili da li je vaša aplikacija zdrava.

Ako ruta vrati status kod 200, vaš servis se smatra zdravim. Kao podsetnik – HTTP status kod 200 znači da je sve u redu!

Ako ne odgovori, orkestrator može automatski da ponovo pokrene uslugu.

I šta je "graciošno gašenje"?

Elegantno gašenje je ono što se dešava kada vaša aplikacija treba da prestane, kao što je slučaj kada se server ponovo pokreće ili kada se odvija implementacija (deployment).

Umesto da odmah prekinete veze, date aplikaciji vremena da završi ono što radi.

U Expressu, možete slušati za `SIGTERM`, što je sistemski signal poslat kada proces treba da se prekine.

Evo primera:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Ovo govori Expressu da prestane sa prihvatanjem novih konekcija i da sačeka da postojeće završe pre nego što izađe.

Takođe treba da obradite `SIGINT`, što je signal poslat kada pritisnete Ctrl+C za zaustavljanje vaše aplikacije tokom lokalnog razvoja.

Iako se `SIGTERM` obično koristi od strane orkestratora kao što je Kubernetes u produkciji, `SIGINT` pokriva svakodnevni rad developera. Bez rukovanja tim, pritiskanje Ctrl+C odmah ubija proces, preskačući bilo kakvo čišćenje koje ste postavili.

Pošto bi oba signala trebalo da pokrenu isto ponašanje gracioznog gašenja, programeri obično slušaju za oba.

Sada, hajde da sve sastavimo

Zamislite da je vaša aplikacija pokrenuta u kontejneru. Kubernetes šalje ``SIGTERM`` da ga ugasi zbog ažuriranja.

Samo osnovnim gracioznim gašenjem, vaša aplikacija prestaje da prima nove veze i završava obrađivanje trenutnih zahteva. Kada je sav posao završen, odspaja se od baze podataka i izlazi čistim putem. Nema izgubljenih podataka, nema oštećenog stanja.

Ali postoji problem — balanser opterećenja ne zna da vaša aplikacija gašenje, pa može nastaviti da vam šalje novi saobraćaj. Tamo je gde dolazi dodatni obrazac ispod.

Sada, imam mali bonus za vas. Pogledaćemo "Označavanje aplikacije kao nezdrave pre gašenja".

Pre gašenja, možete ažurirati vašu proveru zdravlja da vrati status 503. Na taj način, balanser opterećenja vidi nezdrav odgovor i zaustavlja usmeravanje novog saobraćaja na vašu aplikaciju tokom prozora gašenja, kao što je demonstrirano u kodu iznad.

Podsetnik: Status kod `503` znači "Service Unavailable". signalizira da je server privremeno neup sposoban za obradu zahteva, što se tačno dešava tokom gracioznog gašenja.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
