# --description--

Jeste li se ikada pitali šta se dešava kada nešto krene po krivom u vašoj Express aplikaciji?

Šta ako dođe do problema sa konekcijom baze podataka? Ili korisnik poseti rutu koja ne postoji?

Razumevanje kako funkcioniše rukovanje greškama u Expressu je kritično ako želite da izgradite sigurne i pouzdane veb aplikacije.

U ovoj lekciji, proći ćemo kroz to kako Express rukuje greškama, i kako ih možete uhvatiti i na njih odgovoriti koristeći stvarne kodove primere.

## Kako rukovati jednostavnim greškama u Expressu

U Express-u, obrada grešaka je bazirana na middleware-u.

Slično kao redovne middleware funkcije, middleware za rukovanje greškama ima pristup `req`, `res`, i `next`. Ali uključuje jedan dodatni parametar na početku: `err`.

Evo kako izgleda sintaksa:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Definišete ovaj middleware na kraju vašeg steka middlewarea.

Express zna da je rukovaoca grešaka jer ima četiri parametra — novi, ``err``, pored uobičajenih ``req``, ``res``, i ``next`` parametara za druge middleware funkcije.

U sledećoj lekciji, saznaćete više o middlewareu za rukovanje greškama i kako se on uklapa u širi Express middleware stek.

## Kako rukovati asinhronim greškama

Upravljanje asinhronim greškama zavisi od koje verzije Expressa koristite.

Napomena da, bez obzira koju verziju Express-a koristite, i dalje ćete morati da definišete svoj middleware za rukovanje greškama na kraju vašeg steka, nakon svih vaših ruta.

Evo ponovo rukovaoca greškama:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Ekspres 4

U Expressu 4 i nižim verzijama, koje još uvek mnogi projekti koriste, asinhroni problemi nisu **automatski uhvaćeni**.

Ako koristite `async`/`await` i nešto se baci unutar `async` rukovatelja rute, Express neće uhvatiti to osim ako mu ga ručno ne prosledite na `next()`.

Evo kako pravilno rukovati asinhronim greškama u Expressu 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Ako zaboravite blok `try/catch` i `next(err)`, greška može:

* Pokvarenje vaše aplikacije
* Ili rezultirati upozorenjem o neobradioj odbijanju obećanja

Zato je umotavanje asinhroni logike u `try`/`catch` neophodno u Express 4.

Neki programeri čak kreiraju pomoćnu funkciju kako bi izbegli ponavljanje `try/catch` svuda:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Zatim ga koristite ovako:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

Ovo omogućava da ne morate koristiti `try/catch` blokove u svim vašim `async` rutama, a istovremeno i dalje pravilno prosleđujući greške.

## Ekspres 5

Trenutna verzija Express poboljšava ovo ponašanje.

Od Expressa 5, ako ruta `async` baci grešku ili vrati odbijenu obećanje (rejected promise), Express automatski to prosleđuje na middleware za rukovanje greškama.

Što znači da ovaj primer rukuje greškama bez `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Ako ``getUserFromDatabase()`` baci grešku, Express 5 će je uhvatiti i automatski poslati vašem rukovaocu grešaka.

Nije potreban ručni poziv `next(err)`.

## Kako rukovati greškama 404

Greške 404 su malo drugačije. One nastaju kada nijedan put ne odgovara zahtevu.

Možete ih obraditi sa ovim middleware-om:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Važno**: Ovo bi trebalo da ide nakon vaših ruta, ali pre vašeg middlewarea za rukovanje greškama.

Konačno, evo nekih najboljih praksi za rukovanje greškama u Expressu:

* Uvek postavite vaš middleware za rukovanje greškama na kraj.
* Rukujte sa 404 greške odvojeno.
* Koristite `next(err)` da propagirate greške iz ruta ili logiku `async` u Expressu 4 i nižim.

U produkciji, nemojte slati stack trace-ove korisnicima. Zabeležite detalje interni i umesto toga vratite korisnički prijateljska poruka.

Upravljanje greškama u Express-u se tiče podešavanja middleware-a koji prati probleme i odgovara na jasan, siguran način. Bilo da je u pitanju greška u vašem kodu ili nedostajuća ruta, vaša aplikacija nikada ne bi trebalo da se sruši bez plana za popravak.
