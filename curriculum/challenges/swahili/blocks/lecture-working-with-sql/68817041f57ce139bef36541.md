# --description--

SQL injekcija je web sigurnosna ranjivost u kojoj napadači menjaju ili iskorišćavaju upite koje web aplikacija šalje bazi podataka. Na primer, napadači mogu umetnuti zlonamenski SQL kod u upit kako bi pokrenuli taj zlonamenski kod na backendu.

Ovo se može dogoditi kada korisnicima date opciju da unose podatke putem formulara za prijavu, traka za pretragu ili URL parametara, a zatim kada se taj unos korisnika koristi u interpolaciji niza ili konkatenaciji radi generisanja finalne SQL komande.

U zavisnosti od cilja napadača, zlonamenski SQL kod može izazvati značajne poremećaje na vašoj bazi podataka i web aplikaciji, kao što su zaobilazak autentifikacije, krađa podataka iz baze podataka ili modifikovanje/brisanje zapisa u bazi podataka. Stoga, ovu pretnju treba shvatiti veoma ozbiljno.

Na primer, recimo da vaša web aplikacija koristi SQL upit sličan ovom za autentifikaciju korisnika, direktno ubacujući korisničko ime i lozinku koje je korisnik unio:

```sql
'SELECT * FROM users WHERE username = ' + username_input + ' AND password = ' + password_input + ';'
```

Ako napadač u polje `username` unese nešto poput ovoga:

```sql
'" " OR "1"="1" --'
```

I bilo šta drugo u polje `password`, pošto je `"1"="1"` uvek `true` i `--` komentariše ostatak upita, klauzula `WHERE` postaje:

```sql
WHERE username = " " OR TRUE
```

Što je uvek `true`.

U zavisnosti od implementacije baze podataka, ovo može vratiti podatke prvog korisnika ili bilo kog korisnika u bazi podataka i omogućiti napadaču da se prijavi bez validnih kredencijala.

Dakle, kako to sprečiti? Postoje dva glavna načina za sprečavanje SQL injekcije:

- Ne pišite dinamičke upite sa konkatenacijom niza.
- I sprečite da zlonamenski SQL unos bude uključen u upite koji će se izvršiti na backendu.

U praksi, možete koristiti parametrizovane izjave, poznate i kao parametrizovani upiti. Oni razdvajaju SQL strukturu od podataka koje je korisnik unio, što sprečava bazu podataka da tumači unos podatka kao izvršivi kod.

Druga važna tehnika je validacija unosa. Preporučuje se kao sekundarna sigurnosna mera u svim slučajevima, ali je posebno važna za delove SQL upita koji su vezani za imena tabela, imena kolona i redosleđje sortiranja. Ako je moguće, preporučuje se da imena tabela i imena kolona dodelite direktno unutar vašeg koda, a ne iz unosa korisnika.

Ako, uprkos svim preventivnim merama, doživite napad SQL injekcijom, možete ublažiti njegov uticaj na bazu podataka ograničavanjem dozvola svakog naloga u bazi podataka. Dodelite svakom korisniku baze podataka samo dozvole potrebne za njegove zadatke, ali ništa više. Ako račun treba samo da čita podatke iz baze podataka, dodelite mu samo pristup čitanja. I kao preventivna mera, nikada ne dodeljujte administratorske dozvole vašim aplikacionim nalogima. Vrlo je opasno to učiniti jer bi napadač mogao dobiti potpuni pristup i potpunu kontrolu nad bazom podataka.

SQL injekcija je veoma čest napad o kojem vi, kao programer, trebalo da znate i sprečavate u implementaciji vaše web aplikacije. Ovim tehnikama možete zaštititi svoju bazu podataka i zadržati podatke vaših korisnika bezbednim.
