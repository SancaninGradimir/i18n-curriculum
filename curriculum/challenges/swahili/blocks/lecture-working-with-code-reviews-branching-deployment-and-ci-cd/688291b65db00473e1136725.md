# --description--

CI znači Kontinuirana integracija, a CD znači Kontinuirano dostavljanje ili Kontinuirano razmenjivanje. Zajednički, ovi procesi vam omogućavaju da automatski implementirate promene u vašoj aplikaciji nakon osiguravanja da su te promene funkcionalne.

U CI cejlinu, možda pokrenete svoj linter i testove protiv koda. CI cejlinovi se obično pokreću na zahtevima za povlačenje kako bi osigurali da promene nisu slomile aplikaciju, a takođe i na grani `main` kako bi osigurali da je spremno za deployment.

CD cevovod takođe može da radi na zahtevima za povlačenje (pull requests), ako postavite predgledno okruženje kako biste uživo videli promene. Ali takođe ćete ga pokrenuti na vašoj `main` grani kako biste kontinuirano implementirali promene kako se prihvate i spaje.

Ali kako ovo zapravo radi sa GitHub Actions-ima? Prvo, moramo razumeti šta su zapravo GitHub Actions. GitHub Actions pruža efemernih runner-a (zamislite privremap računar) koji vam omogućava da izvršavate specifične komande.

Akcije su konfigurisane sa YAML fajlom. Možete imati više fajlova za specifikovanje različitih akcija koje treba pokrenuti. Na primer, možda imate jedan fajl za vaš CI i drugi za vaš CD.

Pogledajmo osnovni CI primer:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

Ovo je mnogo, pa hajde da ga razložimo. Gledajući prvi deo:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

`name` definiše ime same akcije. Ovo se pojavljuje u oblastima kao što je sekcija provera stanja za PR. `on` svojstva definišu ono što se naziva "okidači radnog toka" - to su događaji koji će pokrenuti vašu akciju.

U ovom primeru, pokrećemo akciju kada se commiti gurnu na `main`, i kada zahtev za povlačenje cilja `main`. To znači da se CI pokreće kada neko napravi zahtev za povlačenje (pull request), i ponovo kada ga spoji(mo).

Sekcija `jobs` je mesto gde definišete šta vaša akcija zapravo radi. Ključ `lint:` je proizvoljni ključ koji definiše interni naziv za taj posao. Svojstvo `name` definiše eksterni naziv za ovaj konkretan posao. A svojstvo `runs-on` definiše u kakvom okruženju bi vaša akcija trebalo da radi - u ovom slučaju, pokrećemo ga u najnovijoj podržanoj verziji Ubuntua.

Svojstvo ``steps`` je mesto gde se čudo dešava. Ovaj blok definiše stvarne korake koje vaša akcija treba da preduzme. Pogledajmo prvi blok tamo:

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

Ovaj korak ima dve svojstva, `name` koja bi korak trebalo da ima, i svojstvo `uses`. Ovo svojstvo `uses` je posebno, jer govori pokretaču akcije da ode i preuzme javnu akciju sa GitHub-a i izvrši je, umesto da pokrene komandu. Akcija `actions/checkout` upravlja kloniranjem i podešavanjem repozitorijuma unutar vašeg pokretača (runner-a).

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

Sledećih dva bloka takođe koriste akcije treće strane. Ovi koraci postavljaju Node i pnpm menadžer paketa. Oba imaju ključ ``with``, koji možete koristiti za prosleđivanje argumenata tim akcijama treće strane. Razmislite o ovome kao o prosleđivanju argumenata funkciji.

Poslednja četiri bloka su mesto gde se stvarno dešava magija.

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

Imamo blok ``Install Dependencies``, koji instalira npm pakete. Naš blok ``Lint Source Files`` pokreće linter, blok ``Verify Build`` osigurava da se TypeScript kod može kompajlirati, a blok ``Run Tests`` pokreće unit testove.

Važno je znati o akcijama da, podrazumevano, ako jedan od ovih koraka ne uspe, preostali će biti preskočeni i ceo rad će biti označen kao neuspeh.

Gde stavljate svoje YAML fajlove kako biste pokrenuli ove akcije? Moraju ići u direktorijum ``.github/workflows`` u vašem repozitorijumu da bi ih GitHub pronašao i koristio.

Postoje mnogo dodatnih stvari koje akcije mogu da urade, i podstičem vas da pročitate dokumentaciju i istražite postojeće akcije u projektima kao što je repozitorijum freeCodeCamp. Ali za sada, trebalo bi da imate pristojan razumevanje osnova GitHub Akcija i CI/CD!
