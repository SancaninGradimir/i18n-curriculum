## Serveri i Klijent-Server Model

- **Server**: Računar ili program koji prima zahteve od drugih računara i šalje podatke ili usluge nazad preko mreže. Termin se odnosi kako na fizički hardver, tako i na softver koji radi na njemu.
- **Client-Server Model**: Distribuirana arhitektura u kojoj klijenti šalju zahteve, a serveri obrađuju te zahteve i vraćaju odgovore.
- **Client**: Proces ili aplikacija (na primer, web pretraživač) koja inicira zahteve ka serveru.
- **Request-Response Cycle**: Klijent šalje zahtev; server ga obrađuje i vraća odgovor.

### Tipovi servera

- **Web serveri**: Isporučuju web sadržaj (HTML, images, CSS) pregledačima.
- **Application serveri**: Obrađuju poslovnu logiku i obrađuju unos korisnika.
- **Database serveri**: Skladište, dohvaćaju i upravljaju podacima.
- **Mail serveri**: Šalju, primaju i skladište e-poštu.
- **File serveri**: Upravljaju skladištenjem i deljenjem fajlova preko mreže.
- **Proxy serveri**: Deluju kao posrednici između klijenta i drugih servera.

## DNS (Sistem imena domena)

- **DNS**: Hijerarhijski i decentralizovani sistem imenovanja koji prevodi domen imena čitljiva za ljude u IP adrese, često nazivan "telefonbook" interneta.
- **IP Address**: Jedinstveni numerički identifikator uređaja na mreži (na primer, `127.0.0.1` u IPv4, `0:0:0:0:0:0:0:1` u IPv6).
- **Domain Name Hierarchy** (čitanje od desna ka levo):
  - **TLD (Top-Level Domain)**: Najdesniji segment (na primer, `.org`, `.com`).
  - **SLD (Second-Level Domain)**: Registrovano ime (na primer, `freecodecamp` u `freecodecamp.org`).
  - **Subdomain**: Identifikuje sekciju glavnog domena (na primer, `www`, `blog`, `api`).

### Proces rešavanja DNS-a

1. Pregledač šalje domen ime **Recursive Resolveru** (obično upravlja ga ISP).
2. Recursive Resolver upita **Root Servere**, koji ga usmeravaju na relevantni TLD server.
3. **TLD Server** vraća adresu **Authoritative Name Servera** za taj domen.
4. Authoritative Name Server vraća **DNS Record** koji sadrži IP adresu.
5. Recursive Resolver isporučuje IP adresu klijentu i **kešira** zapis kako bi ubrzao buduće pretrage.

## TCP/IP

- **Protokol**: Skup pravila i standarda koji omogućavaju uređajima da komuniciraju preko mreže.
- **TCP/IP**: Paket protokola koji određuje kako se podaci pakuju, šalju, rutiraju i primaju.
  - **IP (Internet Protocol)**: Rutira pakete podataka od izvora do odredišta koristeći IP adrese. Nema veze (connectionless) — svaki paket putuje nezavisno, bez garancije dostave.
  - **TCP (Transmission Control Protocol)**: Razdvaja podatke na numerisane segmente, sastavlja ih na odredištu, proverava greške i izgubljene pakete, te zahteva ponovno slanje ako je potrebno. Osigurava pouzdano, usmereno dostavljanje.

## HTTP (Protokol za prenos hiperteksta)

- **HTTP**: Aplikacioni protokol koji definiše pravila za komunikaciju između klijenata i servera na vebu.

- **HTTPS**: Bezbednija verzija HTTP-a gde se podaci šifruju korišćenjem SSL/TLS pre prenosa.

### Struktura HTTP zahteva

- **Request line**: Linija zahteva: `METHOD /path HTTP/version` (na primer, `GET / HTTP/1.1`)
- **Headers**: Zaglavlja: Parovi ključ-vrednost metapodataka (na primer, `Content-Type: application/json`).
- **Body** (opciono): Telo: Podaci poslati na server, korišćeni sa `POST`, `PUT` i `PATCH`.

**Uobičajene HTTP metode:**

| Metoda | Svrha |
| --- | --- |
| `GET` | Dohvati podatke |
| `POST` | Pošalji nove podatke |
| `PUT` | Zameniti resurs |
| `PATCH` | Delimično ažurirati resurs |
| `DELETE` | Ukloniti resurs |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Struktura HTTP odgovora

- **Linija stanja**: Verzija HTTP-a, kod stanja i fraza razloga (na primer, `HTTP/1.1 200 OK`).
- **Zaglavlja**: Metapodaci o odgovoru (na primer, `Content-Type: text/html`).
- **Telo** (opciono): Vraćeni sadržaj ili objašnjenje greške.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## HTTP Status Kodovi

Status kodovi su grupisani po njihovu prvu cifru:

- **1xx** — Informacioni odgovori.
- **2xx** — Uspešni odgovori.
- **3xx** — Poruke preusmeravanja.
- **4xx** — Odgovori o greškama klijenta.
- **5xx** — Odgovori o greškama servera.

### Uobičajeni kodovi

| Code | Name | Meaning |
| --- | --- | --- |
| `200` | OK | Zahtev je uspeo; sadržaj je vraćen. |
| `201` | Created | Resurs je uspešno kreiran. |
| `204` | No Content | Uspeh, ali nema sadržaja za vraćanje. |
| `301` | Moved Permanently | Resurs je trajno premešten na novu URL adresu. |
| `302` | Found | Resurs je privremeno premešten na drugu URL adresu. |
| `304` | Not Modified | Resurs nije promenjen od poslednjeg zahteva; koristite kešovanu verziju. |
| `400` | Bad Request | Server nije mogao da razume zahtev (nepravilna sintaksa ili podaci). |
| `401` | Unauthorized | Potrebni su kredencijali za autentifikaciju. |
| `403` | Forbidden | Klijent nema dozvolu za resurs, čak ni sa važećim akreditivima. |
| `404` | Not Found | Resurs ne postoji na serveru. |
| `500` | Internal Server Error | Neočekivana greška na serveru. |
| `502` | Bad Gateway | Server je primio nevažeći odgovor od upstream servera. |
| `503` | Service Unavailable | Server privremeno nije u stanju da obradi zahteve. |

## Model HTTP Zahtev-Odgovor

Ciklus zahtev-odgovor sastoji se od ovih komponenti koje rade zajedno:

- **Klijent**: Pretraživač (Browser), mobilna aplikacija ili API klijent koji šalje HTTP zahteve.
- **HTTP Method**: Akcija koja se izvršava (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: Adresa resursa koji se traži.
- **Headers**: Metadata za zahtev ili odgovor (na primer, `Authorization`, `Content-Type`).
- **Body**: Podaci poslati sa zahteva `POST`, `PUT`, `PATCH` ili `DELETE` (često JSON ili podaci forme).
- **Server**: Back-end sistem koji obrađuje zahteve i vraća odgovore.
- **Status Code**: Numerički kod koji ukazuje na uspeh, grešku ili preusmeravanje.
- **Response Body**: Stvarni povratni podaci, obično HTML, JSON ili XML.

### Šest koraka ciklusa

1. **Klijent inicira zahtev** sa metodom, URL-om, zaglavljima i opcionalnim telom (body).
2. **Zahtev putuje do servera** preko interneta.
3. **Server obrađuje zahtev**: proverava metodu, URL, autentifikaciju i upita bazu podataka ako je potrebno.
4. **Server priprema odgovor** sa statusnim kodom, zaglavljima i opcionalnim telom (body).
5. **Odgovor se vraća** klijentu.
6. **Klijent obrađuje odgovor**: renderuje podatke u slučaju uspeha (2xx) ili prikazuje poruku o grešci u slučaju neuspeha (4xx/5xx).

## Pošiljanje HTML formulara

- Forme prikupljaju i šalju podatke korisnika koristeći element ``<form>`` sa elementima ``<input>``.
- Svaki unos zahteva atribut ``name`` kako bi server mogao da identifikuje podatke.
- **Validacija na klijentskoj strani**: Koristite atribute ``required`` i ``pattern`` za trenutnu povratnu informaciju (feedback).
- **Validacija na serveru**: Uvek validirajte na serveru radi sigurnosti, bez obzira na provere na klijentskoj strani.
- Atribut ``action`` određuje gde se podaci forme šalju.
- Atribut ``method`` određuje kako se oni šalju (``GET`` ili ``POST``; za ``PUT``, ``PATCH``, i ``DELETE`` je potrebna privremena rešenja (workarounds) kao što su Fetch API ili ``method-override`` u Express-u).

### Proces podnošenja forme

1. Korisnik popunjava i šalje formular.
2. Pregledač pokreće validaciju na strani klijenta (client-side); nastavlja ako je važeći/validan.
3. Pregledač šalje podatke iz formulara na URL u `action` koristeći navedeni `method`.
4. Server prima i obrađuje podatke (skladištenje u bazi podataka, autentifikacija korisnika, itd.).
5. Server može pokrenuti dodatnu validaciju na strani servera (server-side).
6. Server odgovara porukom o uspehu, greškom ili preusmeravanjem (redirect).
7. Korisnik vidi povratnu informaciju na osnovu odgovora servera.

## Model Web standarda i tijela za standardizaciju

- **Web standards**: Pravila i specifikacije koje pregledači prate za renderovanje stranica i omogućavanje web mogućnosti.
- **WHATWG**: Tijelo standarda koje održava HTML i DOM kao kontinuirano ažurirani "živi standard", vođen od strane glavnih timova za browser engine-e.
- **W3C**: Glavna organizacija standarda, koja nadgleda CSS, SVG, smernice pristupačnosti (WCAG) i brojne web API-je.
- **ECMA / TC39**: ECMA objavljuje ECMAScript specifikaciju (JavaScript standard); TC39 unapređuje jezičke funkcije kroz proces predloga u pet faza (Faza 0–4).
- **Khronos Group**: Odgovorna za WebGL i WebGL 2, koji donose GPU-akcelerovano 3D renderovanje u pregledač.

## Proces kreiranja web standarda

1. **Identifikovanje potrebe**: Razvojnik ili inženjer pregledača dokumentuje stvaran i široko prisutan problem.
2. **Predlog i rano razmatranje**: Ideja se neformalno piše kako bi se prikupio povratni odgovor (feedback) i procenilo interesovanje proizvođača pregledača.
3. **Pisanje specifikacije**: Napravljen je precizan, formalan dokument koji pokriva sve rubne slučajeve, uslove greške i implikacije za bezbednost.
4. **Pregled, prigovor i iteracija**: Nacrt se javno pregleda; proizvođači podižu prigovore i specifikacija se revidira dok se ne postigne konsenzus.
5. **Implementacija i testiranje**: Proizvođači pregledača implementiraju funkciju iza zastavice (flag); zajednički testni set proverava da li sve implementacije ponašaju identično.
6. **Finalizacija**: Kada provere prošdere više nezavisnih implementacija, standard se finalizuje (W3C Recommendation, TC39 Stage 4, ili ažuriranje živoog standarda WHATWG).

## Životni ciklus karakteristika web standarda

1. **Predlog**: Neko identifikuje problem i zapisuje ga kako bi utvrdio da li drugi dele taj isti problem.
2. **Inkubacija**: Ideja se testira kroz otvorenu diskusiju; mnogi predlozi ovde propadaju ako su previše nišni ili ako nijedno rešenje ne zadovoljava sve.
3. **Specifikacija**: Piše se formalna specifikacija koja pretvara apstraktne koncepte u precizne, implementabilne pravila.
4. **Testiranje interoperabilnosti**: W3C Web Platform Tests paketi proveravaju da li svaki pretraživač implementira funkciju identično.
5. **Finalizacija**: Više uspešnih implementacija znači da se funkcija isporučuje korisnicima kao deo osnovne web platforme (baseline web platform).
6. **Deprekacija**: Funkcije koje su zamenjene ili problematične označavaju se za prestanak korišćenja; uklanjanje iz pretraživača je sporo zbog obaveza kompatibilnosti sa starijim verzijama (backwards-compatibility obligations).

## Ključni principi web standarda

- **Otvorenost**: Standardi se razvijaju otvoreno i implementiraju bez naknade od strane bilo koga.
- **Pristupačnost**: Web je dizajniran da bude upotrebljiv za sve, sa pristupačnošću ugrađenom u HTML, CSS i ponašanje pregledača.
- **Interoperabilnost**: Stranica napravljena prema standardu treba da se ponaša identično u svim pregledačima i operativnim sistemima.
- **Kompatibilnost sa starijim verzijama**: Nove mogućnosti se dodaju na postojeće kako bi stare stranice nastavile da rade u modernim pregledačima.
- **Privatnost i bezbednost**: Novi API-ji koji bi mogli da izlože podatke korisnika podvrgnuti su značajnoj pažnji; platforma štiti korisnike po podrazumevanju (by default).
- **Slojevost**: HTML obrađuje strukturu, CSS obrađuje prezentaciju, a JavaScript obrađuje ponašanje — svaki sloj radi nezavisno.
- **Progresivno poboljšanje**: Funkcionalnosti moraju prvo da rade za sve, a zatim se poboljšavaju za korisnike sa sposobnijim pregledačima ili uređajima.
