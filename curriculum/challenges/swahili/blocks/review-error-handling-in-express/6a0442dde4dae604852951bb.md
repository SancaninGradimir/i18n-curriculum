## HTTP Kodovi Statusa Odgovora

- **Kodovi statusa** su kratke numeričke poruke koje obaveštavaju klijenta šta se dogodilo nakon slanja zahteva (na primer, 200, 404 ili 500).

### 1xx — Informativni

- **100 Continue** — server obaveštava klijenta da pošalje ostatak zahteva.
- **101 Switching Protocols** — server menja protokole (često se koristi sa WebSockets protokolom).

### 2xx — Uspeh

- **200 OK** — zahtev je uspešno izvršen.
- **201 Created** — novi resurs je uspešno kreiran.
- **204 No Content** — zahtev je uspešan, ali nema tela odgovora (često se koristi za `DELETE`).

### 3xx — Preusmeravanje

- **301 Moved Permanently** — resurs ima novi trajni URL.
- **302 Found** — resurs je privremeno na drugom URL-u.
- **304 Not Modified** — klijent može da koristi keširanu verziju; ništa se nije promenilo.

### 4xx — Klijentske greške

- **400 Bad Request** — Zahtev je neispravno formatiran ili nije validan.
- **401 Unauthorized** — Potrebna je autentifikacija.
- **403 Forbidden** — Korisnik je autentišovan, ali nema dozvolu (ovlašćenje).
- **404 Not Found** — Zahtevani resurs ne postoji.

### 5xx — Greške servera

- **500 Internal Server Error** — generički kvar na strani servera.
- **502 Bad Gateway** — proxy je primio nevažeći odgovor od drugog servera.
- **503 Service Unavailable** — server je privremeno preopterećen ili nedostupan.

### Korišćenje Status Kodova u Expressu

Status kodovi su ključni za komunikaciju ishoda HTTP zahteva. U Expressu, možete postaviti ove kodove koristeći `res.status()`.

Imajte na umu da su status kodovi standardizovani širom web-a (npr., 200 OK, 404 Not Found). Treba ih koristiti kako biste jasno ukazali na uspešnost ili neuspešnost.

Napišite rukovatelja rute koji vraća JSON objekat sa specifičnim HTTP status kodom kada se korisnički ID ne pronađe. Koristite ``res.status(404)`` i pošaljite odgovarajuću poruku.

- `res.status(400).send("Bad Request")` — šalje 400 sa porukom običnog teksta.

- `res.status(500).send("Internal Server Error")` — šalje 500 sa porukom.

## Rukovanje Greškama u Express-u

- **Middleware za obradu grešaka** ima četiri parametra: `(err, req, res, next)`. Četvorostruk potpis je ono što signalizira Expressu da se radi o rukovaocu grešaka (error handler).
- **Postavite middleware za obradu grešaka na kraj steka middleware-a, nakon svih ruta.**

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Express 4 Asinhrono Upravljanje Greškama

- Asinhroni problemi/greške nisu **automatski uhvaćeni** u Expressu 4.
- Omotajte asinhronu logiku u `try/catch` i pozovite `next(err)` da prosledite grešku (error) rukovaocu grešaka (error handler).

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

Omotač ``asyncHandler`` izbegava ponavljanje ``try/catch`` u svakoj ruti:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Rukovanje greškama 404

Dodajte catch-all middleware **nakon ruta, ali pre error handler-a** za obradu neusaglašenih ruti:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Najbolje prakse za obradu grešaka

* Middleware za rukovanje greškama uvek postavite na sam kraj.
* Rukujte sa 404 greškama odvojeno od drugih grešaka.
* U produkciji, interni detalje o greškama logujte — nikada ne šaljite stack trace-ove korisnicima.

## Debagovanje i logovanje u Express-u

- **Ugrađeni debugging**: postavite okolnu varijablu ``DEBUG`` da biste omogućili ugrađeni debug izlaz Express-a.
  - ``DEBUG=express:* node index.js`` — omogućava sve poruke za debagovanje (debugging) Express-a.
  - ``DEBUG=express:router node index.js`` — prikazuje samo izlaz vezan za rutere (router).
  - PowerShell: ``$env:DEBUG = "express:*"; node index.js``
  - CMD: ``set DEBUG=express:* && node index.js``
- **``morgan``**: popularni middleware za logovanje HTTP zahteva (metoda, URL, status kod, vreme odgovora).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **Debugovanje** prati kako Express obrađuje zahteve i middleware bez zagađivanja logova produkcije.
- **Logovanje** beleži šta se dešava tokom zahteva — korisno za otkrivanje grešaka (bugova) i razumevanje protoka saobraćaja.

## Provere zdravlja i kontrolisano gašenje

### Provera zdravlja

* **Ruta za proveru zdravlja** (health check route) izveštava da li aplikacija radi ispravno.
* Load balanceri i orkestratori (na primer Kubernetes, AWS Elastic Beanstalk) pinge ovu rutu kako bi utvrdili da li je servis zdrav.
* Odgovor **200** znači da je sve u redu; neizostajanje odgovora pokreće automatski restart.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### Kontrolisano zaustavljanje

- **Blag gašenje** (graceful shutdown) omogućava aplikaciji da završi sve postojeće zahteve pre nego što prestane, umesto da odmah prekine veze.
- Slušajte signal `SIGTERM` (koji je poslat kada proces treba da se zaustavi/terminira) i pozovite `server.close()`:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

Pratite signal ``SIGINT`` (poslat kada programer pritisne Ctrl+C tokom lokalnog razvoja) i pozovite istu logiku za gašenje kao ``SIGTERM``.

### Označavanje aplikacije kao nezdrave pre gašenja

Tokom gašenja, vratite status kod **503** iz provere zdravlja kako bi load balancer prekinuo usmeravanje novog saobraćaja ka aplikaciji.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
